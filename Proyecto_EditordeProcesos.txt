Editor y visualizador de procesos¶ 
El proyecto consiste en desarrollar un Sistema de Gestión de 
Procesos multiempresa que permita a cada organización registrarse, crear su propio 
espacio independiente y administrar usuarios con distintos roles (administrador, 
editor y solo lectura). El sistema debe garantizar autenticación segura, control de 
acceso por empresa y separación de la información, asegurando que los procesos 
pertenezcan a la empresa y no a usuarios individuales. 
La solución permitirá crear, editar, consultar y eliminar procesos organizacionales, 
registrando información como nombre, descripción, categoría y estado (borrador o 
publicado). Además, debe mantener historial de cambios y trazabilidad, permitir 
búsquedas y filtros, y manejar eliminaciones lógicas (estado inactivo) para no perder 
información histórica. Todo el modelo debe estar orientado a mantener consistencia 
y control sobre la evolución de los procesos. 
Finalmente, el sistema debe permitir el modelado estructurado de los procesos 
mediante actividades, arcos y gateways (decisiones). Las actividades estarán 
asociadas a roles funcionales definidos por la empresa, los arcos representarán la 
secuencia del flujo y los gateways permitirán ramificaciones exclusivas, paralelas o 
inclusivas. También se deberá gestionar la creación, edición, consulta y eliminación 
de roles, validando que no estén en uso antes de eliminarlos y asegurando 
coherencia en todo el diagrama del proceso. 
Índice de Historias de Usuario¶ 
HU-01 · Registro de empresa¶ 
Resumen 
Como representante de una empresa, quiero crear una cuenta empresarial para que 
mi organización tenga un espacio propio donde administrar sus procesos. 
Criterios de aceptación 
• 
Se solicita nombre de empresa, NIT (u otro identificador), correo de contacto. 
• 
Se genera un usuario administrador inicial. 
• 
La empresa queda registrada como entidad independiente. 
HU-02 · Registro de usuario en empresa¶ 
Resumen 
Como administrador de empresa, quiero crear y gestionar cuentas de usuario dentro 
de mi empresa para que diferentes colaboradores puedan aportar a la 
documentación de procesos. 
Criterios de aceptación 
• 
Puedo invitar usuarios por correo electrónico. 
• 
Puedo asignar roles (ej: administrador, editor, solo lectura). 
• 
Los procesos creados pertenecen a la empresa, no al usuario.

Acceso restringido a la empresa a la que pertenezco.

HU-21 · Configurar pool por empresa¶ 
Resumen 
Como administrador de la plataforma, quiero que cada empresa tenga su propio pool 
(área de procesos) para mantener aislados los procesos y recursos entre 
organizaciones, de modo que no se mezclen datos ni procesos entre empresas. 
Criterios de aceptación 
• 
Existe una opción al crear/editar empresa para crear o asignar 
un pool asociado. 
• 
Los procesos creados dentro de una empresa quedan asociados por defecto 
al pool de esa empresa. 
• 
Usuarios de una empresa sólo ven y gestionan procesos pertenecientes al 
pool de su empresa (se respeta separación multiempresa). 
• 
La creación/edición de pools queda limitada a roles con permiso (ej. 
administrador de empresa). 
Notas 
• 
Implementar validaciones para evitar asignar un proceso al pool de otra 
empresa. 
• 
Registrar auditoría de creación/actualización del pool. 
HU-22 · Diferenciar pool y lane (swimlane)¶ 
Resumen 
Como usuario que modela procesos, quiero distinguir entre pool (espacio 
organizativo) y lane/swimlane (roles o áreas dentro de un pool) para poder representar 
responsabilidades y responsabilidades por rol dentro de cada empresa. 
Criterios de aceptación 
• 
La interfaz permite crear lanes dentro de un pool para reflejar roles o 
departamentos. 
• 
Al asignar actividades, se puede seleccionar la lane responsable (y por tanto el 
rol encargado). 
• 
Las lanes se muestran visualmente dentro del diagrama de proceso y su 
persistencia queda ligada al pool. 
• 
Las reglas de visibilidad siguen la separación por pool; usuarios sólo pueden 
ver lanes dentro de su pool. 
Notas 
• 
Documentar la diferencia conceptual entre pool (borde de proceso) y lane 
(organización interna por responsabilidades). 
HU-23 · Compartir procesos entre pools (alcance y límites)¶ 
Resumen 
Como arquitecto de procesos, quiero establecer reglas claras para compartir o no 
compartir procesos entre pools (empresas), definiendo permisos y límites, para 
soportar escenarios de colaboración controlada sin romper el aislamiento de datos. 
Criterios de aceptación 
• 
Existe una política configurable que define si un proceso puede marcarse 
como compartido y qué pools pueden acceder a él. 
• 
Solo usuarios con permiso global (o administrador) pueden marcar un proceso 
como compartido. 
• 
El acceso a procesos compartidos respeta permisos: sólo lectura o 
lectura/edición según rol asignado. 
• 
Se registra auditoría cuando un proceso cambia su estado de 
visibilidad/compartición. 
Notas 
• 
Por defecto, los procesos no son compartidos para preservar separación 
multiempresa. 
HU-24 · Asociar roles y permisos a un pool¶ 
Resumen 
Como administrador de empresa, quiero asociar roles (por ejemplo administrador, 
editor, lector) al pool de mi empresa y definir permisos por rol, para controlar qué 
acciones pueden realizar los usuarios sobre los procesos dentro del pool. 
Criterios de aceptación 
• 
Existe una gestión de roles a nivel de pool donde crear/editar/eliminar roles 
específicos de la empresa. 
• 
Cada rol tiene permisos configurables (crear proceso, editar, eliminar, 
publicar, gestionar roles). 
• 
Los usuarios se asignan a roles dentro del pool y obtienen permisos en 
consecuencia. 
• 
Las acciones en la UI y en la API respetan los permisos definidos por rol. 
Notas 
• 
Evitar que la eliminación de un rol deje procesos sin propietario; requerir 
reasignación previa. 
HU-25 · Enviar mensaje entre procesos (Message Throw)¶ 
Resumen 
Como diseñador de procesos, quiero poder lanzar (throw) un message desde un 
proceso para notificar o activar otra instancia de proceso, de modo que se puedan 
orquestar flujos distribuidos entre procesos independientes. 
Criterios de aceptación 
• 
Existe una tarea o evento de tipo Message Throw que permite definir el nombre 
del mensaje y los datos payload. 
• 
Otro proceso con un Message Catch configurado con el mismo nombre puede 
iniciarse o recibir el mensaje en tiempo de ejecución. 
• 
El envío incluye payload (JSON) opcional que puede mapearse a variables del 
proceso receptor. 
• 
Se registran eventos de envío y recepción para trazabilidad. 
Notas 
• 
Definir claramente el comportamiento cuando no hay receptor: fallback 
(guardar, reintentar o error). 
HU-26 · Envío de notificaciones externas (mensaje a sistema externo)¶ 
Resumen 
Como integrador, quiero que un proceso pueda enviar mensajes a sistemas externos 
(ej. correo, colas, APIs) para notificar eventos o invocar servicios fuera del motor de 
procesos. 
Criterios de aceptación 
• 
Existe una tarea de integración que envía un mensaje a un destino externo 
configurable (SMTP, webhook, broker). 
• 
La tarea admite mapeo de variables del proceso al payload del mensaje. 
• 
Se manejan y registran errores de entrega; existe política de reintento 
configurable. 
• 
Los destinos requieren credenciales/secretos seguros gestionados por la 
configuración del entorno. 
Notas 
• 
Separar conectores por tipo (email, webhook, queue) y exponer adaptadores 
para facilitar pruebas. 
HU-27 · Recibir mensaje y activar proceso (Message Catch)¶ 
Resumen 
Como diseñador de procesos, quiero que un proceso pueda tener un Message 
Catch que reciba mensajes externos o de otros procesos y ejecute actividades en 
respuesta, de modo que el motor pueda reaccionar a eventos asíncronos. 
Criterios de aceptación 
• 
Se pueden definir Message Catch en el diagrama con el nombre del mensaje 
esperado. 
• 
Cuando llega un mensaje con el nombre y correlación adecuados, se activa la 
instancia de proceso o se continúa la ejecución en la instancia existente. 
• 
El mensaje entrante puede mapearse a variables del proceso receptor. 
• 
El sistema valida la firma o credenciales del emisor cuando el mensaje 
proviene de una fuente externa. 
Notas 
• 
Implementar pautas de seguridad y validación para evitar inyección de datos 
por mensajes externos. 
HU-28 · Correlación de mensajes con instancias de proceso¶ 
Resumen 
Como operador de procesos, quiero que los mensajes entrantes se correlacionen 
correctamente con instancias de proceso (por ejemplo usando un businessKey o 
variables), para asegurar que la información alcance la instancia correcta y no inicie 
procesos duplicados. 
Criterios de aceptación 
• 
Existe una regla de correlación configurable (por businessKey, expresión o 
variable de correlación) usada por Message Catch. 
• 
Si la correlación encuentra una instancia en espera, el mensaje se entrega a 
esa instancia; si no, se puede iniciar una nueva instancia si está configurado. 
• 
Se registran eventos de correlación y casos no correlacionados para 
diagnóstico. 
• 
En caso de múltiples coincidencias, se aplica una política definida (error, 
entregar a la primera, o crear nueva instancia). 
Notas 
• 
Probar escenarios de correlación con datos de alta concurrencia para evitar 
condiciones de carrera.
